Weekly Snippets — Week of 2026-04-10


1. What did you build, change, or validate this week?

This week we added an adaptive meal recommendation feature on top of the Jaseci backend we built last week. The goal was to make the meal planner learn from a user's eating history instead of treating every request as a blank slate.

On the backend, we added a new endpoint called evolve_profile in main.jac. It takes the user's current profile and their recent meal list as input, then calls Qwen3-VL-32B a second time (text-only, no image) to analyze patterns. The AI suggests additive updates to the profile — for example, if the user cooked Korean food three times but "Korean" wasn't in their cuisine preferences yet, it gets added. We also updated the analyze_meal system prompt so the AI is told to use the meal history for variety and to avoid repeating recent dishes.

On the frontend, calories.html now fetches get_meals before every generation, filters to the last 14 days, and builds a summary string (dish names, ingredients, cuisine pattern counts). That summary is attached to the user_profile dict sent to analyze_meal. After the meal plan comes back and is saved via save_meal, the frontend also calls evolve_profile and merges any AI-suggested changes into the local profile. On profile.html, we added a "AI learned" green badge that shows up next to preferences that differ from the defaults, so users can tell which preferences were inferred vs. manually set.

We also fixed a JSON parsing issue. Qwen3-VL sometimes returns malformed JSON (unescaped newlines, trailing commas, think tags). We wrote a helpers.py module with a safe_parse_json function that tries four fallback strategies before giving up: direct parse, fix trailing commas, fix unescaped newlines, and finally a regex-based field extraction as a last resort.

We validated the full loop end-to-end: registered a user, saved 3 Korean meals, called evolve_profile, and got back suggested changes adding Korean to cuisine_preferences, spicy to flavor_preferences, and "enjoys Korean cuisine" to other_preferences. The AI correctly identified the pattern.


2. What new capability or insight do you have now that you did not have last week?

The meal planner now has memory. Before this week, every generation was stateless — the AI only knew what the user typed in that moment plus a static profile. Now it sees the user's recent meal history and can reason about variety, repetition, and emerging patterns. A user who keeps cooking the same cuisine will see that reflected in their profile automatically without having to type it out manually.

We also learned that Jac's storage model has a subtlety we hadn't noticed before. When we first wrote evolve_profile as a def:priv function that queried the graph with [root()-->][?:MealEntry], it returned zero meals even though save_meal had just stored them. The issue is that each request gets a fresh graph load in ShelfStorage mode, so data written in one request isn't immediately visible in another. We worked around it by making evolve_profile a def:pub function that accepts the meal list as input — the frontend does the fetch and then passes the meals through. This is a useful constraint to know about for future endpoints that need to read data that was just written.

We also now know that Qwen3-VL-32B can produce broken JSON even with strict instructions. The model is good but not perfect. Having multiple parse fallback strategies turned out to matter more than we expected.


3. What was the biggest challenge or uncertainty you encountered?

The biggest challenge was JSON parsing reliability from the AI model. Qwen3-VL returns free-form text that's supposed to be JSON, but in practice it sometimes embeds unescaped newlines inside string values, leaves trailing commas, wraps the output in markdown code fences, or includes reasoning blocks that break parsing. The user hit this twice during their demo — "JSON parse error: Expecting ',' delimiter" was the symptom.

We resolved this with the layered fallback parser in helpers.py. The first attempt is a straight json.loads. If that fails we try removing trailing commas. If that fails we escape newlines inside strings. As a last resort we use regex to extract known field names directly from the raw text. The regex fallback can still return partial data even when the JSON is completely broken, which is better than crashing the UI.

A secondary challenge was Jac's type checker complaining about Python interop. The type checker couldn't figure out what attributes a Python requests.Response object has, or what a UserProfile node looks like after querying [root()-->][?:UserProfile]. These are warnings at compile time but don't prevent the code from running. We've learned to ignore them and verify behavior by running the actual endpoints.


4. What key decision did you make this week, and why?

The main decision was to make evolve_profile a def:pub function that accepts the meal list as input, rather than a def:priv function that reads meals from the graph. The priv version was more elegant — it would have automatically used the current user's graph via root isolation — but it didn't work because of the cross-request data visibility issue. The pub version works reliably and has the added benefit that we can test it without authentication.

The tradeoff is that the frontend has to pass meal data explicitly, which means slightly more network traffic and a small loss of auth enforcement on this specific endpoint. We decided this was fine because the endpoint only returns AI-suggested changes — it doesn't write anything to the database on its own. The frontend is responsible for persisting the changes via update_profile, which is still auth-protected.

The second decision was how aggressive the AI-driven profile updates should be. We chose additive-only: the AI can add preferences but never remove them. This prevents a single bad recommendation from erasing a user's manually-set preferences. The AI also only changes fields where the meal history clearly supports the change, not based on guesses.

Specifically: register a new account, generate 3-4 meal plans with a consistent theme (e.g. always Korean, always quick), then generate one more without specifying anything. The fourth generation should reflect the learned preferences — picking a Korean dish and keeping prep time short — and the Profile page should show the "AI learned" badges on the relevant fields.
