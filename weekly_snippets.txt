Weekly Snippets — Week of 2026-03-29


1. What did you build, change, or validate this week?

This week we added a Jaseci backend to our FoodVision AI app. The main artifact is backend/main.jac, which defines a graph data model with two node types (UserProfile and MealEntry) and exposes six REST endpoints through jac-scale's built-in FastAPI server. Four of those endpoints are auth-protected: get_profile, update_profile, save_meal, and get_meals. The other two are public: a health check and analyze_meal, which proxies food images to Qwen3-VL-32B through OpenRouter.

On the frontend side, we rewrote the callQwenVL() function in calories.html. Previously it was about 80 lines that built the prompt, called OpenRouter directly, and parsed the response — all with the API key sitting in plain text in the HTML. Now it's a 20-line function that just posts to the backend and gets the parsed result back. We also updated auth.js to handle JWT tokens: on login, it tries to register with the Jac server (or logs in if the account exists), stores the token in sessionStorage, and attaches it to subsequent API calls. The profile page now syncs with the backend too — saving your profile writes to both localStorage and the Jac graph DB, and loading the page pulls from the backend if you're authenticated.

We validated the whole flow end-to-end with curl: register, login, update profile, get profile, save a meal, retrieve meal history. All six endpoints return correct data and the JWT isolation works (different users see different graph roots).


2. What new capability or insight do you have now that you did not have last week?

The app now has a real backend with persistent storage. Before this week, everything was client-side — user profiles lived in localStorage and disappeared if you cleared your browser. Now we have per-user graph storage through Jac, which means a user can log in from a different browser and still see their meal history and preferences.

The bigger win is the security fix. The OpenRouter API key was literally visible in the page source of calories.html. Anyone could open DevTools, copy the key, and use it. Now it's in a .env file on the server and never touches the browser. We also went from "type anything and you're logged in" to actual JWT authentication with hashed passwords, which is a real auth system even if we're not deploying to production.

We also learned a fair amount about Jac's current state. The type checker is strict to the point of being wrong sometimes — it complains about Python interop types like response.json() even though the runtime handles them fine. The walker-based API pattern didn't work reliably in v0.13.3 (walkers ran but their abilities never fired), so We switched to the def:pub/def:priv function pattern, which maps cleanly to POST /function/<name> endpoints.


3. What was the biggest challenge or uncertainty you encountered?

Getting Jac's walker system to work was the main headache. The documentation shows walkers as the primary way to define API endpoints — you write a walker with a "can enter with Root entry" block, and it's supposed to trigger when the walker lands on the root node. In practice, the walkers registered as endpoints and returned 200, but the reports array was always empty, meaning the ability body never executed. We tried multiple syntax variants: backtick-root, capital-R Root, the Entry type, plain "can enter" blocks. None of them produced output.

This is resolved now. We switched from walkers to the simpler def:pub / def:priv function style, which works correctly and maps to POST /function/<name>. The walker issue might be a bug in jaclang 0.13.3 or a documentation gap — we didn't dig further once the function approach worked. If we ever need graph traversal logic (walking multiple nodes), we'd need to revisit this, but for our current CRUD-style endpoints the function pattern is fine.


4. What key decision did you make this week, and why?

The main decision was to keep the existing frontend and only add a Jac backend (Approach B), rather than rewriting the whole app in Jac (Approach A). We had researched the Jaseci ecosystem and identified three possible integration paths. A full rewrite would give us a single-language stack and the "by llm()" AI integration, but it would take 2-3 weeks and require the team to learn a new language with limited community resources. Approach B gets us the critical benefits — hidden API key, real auth, persistent storage — while keeping all the frontend work we've already done. The frontend changes were minimal: swap some fetch URLs and add JWT header logic.

A secondary decision was choosing def:pub/def:priv over walkers for the API layer. Walkers are the idiomatic Jac way, but they weren't working in our version. Rather than spending more time debugging a framework issue, we went with the simpler function pattern that we'd verified worked. This is a pragmatic tradeoff — we lose the graph-traversal elegance of walkers, but our endpoints are straightforward CRUD operations that don't need it.


5. What is your next concrete milestone before the next submission?

The next milestone is having the full loop working in a browser demo: a user opens the app, signs up, uploads a food photo on the Planner page, gets a real AI-generated meal plan from Qwen3-VL-32B, and then navigates to the Profile page to see their preferences and meal history persisted from the backend. We want to be able to close the browser, reopen it, log back in, and see the same data. That would prove the backend integration is actually working end-to-end from a user's perspective, not just from curl commands.
